# AI Signal RSS 阅读器设计说明

日期：2026-07-17  
状态：用户已确认设计方向，等待书面设计复核

## 1. 产品目标

制作并上线一个免费托管的 HTML RSS 阅读器，聚合用户提供的 AI、机器学习、技术团队与个人博客信源。文章按发布时间倒序混排，不同来源使用固定颜色区分。网站采用苹果风格与系统字体，桌面端和手机版均可完整使用，并每天自动更新。

## 2. 已确认的产品范围

- 网站名称：AI Signal。
- 界面语言：中文；文章标题与摘要保留信源原文，不自动翻译。
- 首页采用杂志式精选卡片。
- 封面策略：优先使用 RSS 图片，其次使用网页分享图；仍无图片时使用来源专属渐变。
- 点击文章后先打开站内摘要详情，再提供“阅读原文”链接。
- 支持搜索、来源筛选、类别筛选、未读筛选和收藏筛选。
- 支持收藏与已读标记，数据保存在当前浏览器，不需要注册或数据库。
- 支持浅色、深色和跟随系统三种主题；手动选择会保存在当前浏览器。
- 支持桌面、平板和手机布局。
- 免费托管于公开 GitHub 仓库的 GitHub Pages，并使用 GitHub Actions 每天自动更新。

## 3. 不在本期范围内

- 用户账号、跨设备同步和云端数据库。
- 自动翻译、AI 摘要或付费 API。
- 在本站转载完整文章正文。
- 评论、社交分享统计和个性化推荐算法。

## 4. 技术架构

### 4.1 前端

- 使用 React、Vite 和 TypeScript 构建静态单页应用。
- GitHub Pages 只提供构建后的静态文件和预先生成的文章数据。
- 文章详情使用可关闭的模态阅读层；手机端扩展为全屏详情。
- 收藏、已读状态和主题偏好写入 `localStorage`。
- 使用 URL 哈希保存当前筛选和文章详情状态，避免 GitHub Pages 刷新路径出现 404。

### 4.2 数据更新器

- Node.js 更新脚本读取统一的信源配置。
- RSS 或 Atom 地址直接解析；普通博客页面先读取 HTML 中的 feed discovery 链接，再尝试常见 feed 路径。
- 每个信源限制抓取时间，并进行有限重试；全局使用受控并发，避免对网站造成集中压力。
- 标准化标题、原文链接、发布日期、摘要、作者、来源、类别和封面。
- 封面提取顺序：RSS enclosure 或 media 字段、正文首图、网页 Open Graph 图、来源专属渐变。
- 通过规范化链接、GUID 和内容指纹去重。
- 新数据与上一版数据合并，保留最近 90 天且最多 1,000 篇文章。
- 生成文章数据、来源信息和更新状态三个静态 JSON 文件。

### 4.3 每日更新和发布

- GitHub Actions 在每天 00:17 UTC，即香港时间 08:17 运行，避开整点高峰。
- 工作流也支持手动运行，并在主分支有代码变更时构建。
- 每日任务抓取信源、运行数据检查、构建网站并部署 GitHub Pages。
- 生成数据发生变化时提交到仓库，保留更新历史并维持仓库活动。
- 首次上线时创建公开仓库 `ai-signal-reader`，启用 GitHub Pages 的 Actions 发布源。

## 5. 页面与组件

### 5.1 顶部区域

- AI Signal 名称与简洁图形标记。
- 当前日期、最近更新时间和本次异常来源数量。
- 搜索框。
- “全部 / 未读 / 收藏”分段切换。
- 来源、类别与主题设置按钮。

### 5.2 杂志式文章流

- 数据和键盘浏览顺序严格按照发布时间倒序。
- 最新文章在视觉上使用较大的头条卡片，其余使用响应式卡片网格。
- 卡片显示来源、相对时间、标题、短摘要和封面。
- 每个来源拥有稳定且可区分的颜色；颜色用于来源标签、缺图渐变和细节强调。
- 已读文章降低视觉强调，但仍保持足够对比度。
- 没有发布日期的文章标记“日期未知”，排在具有有效日期的文章之后。

### 5.3 文章详情

- 显示来源、发布日期、标题、作者和经过清理的摘要。
- 提供收藏切换、关闭和“阅读原文”按钮。
- 打开详情时标记已读；用户可再次切换为未读。
- 原文在新标签页打开，并使用安全的外部链接属性。

### 5.4 筛选与空状态

- 搜索标题、摘要、作者和来源名称。
- 可按单个来源或类别筛选。
- 无结果、首次加载失败、全部已读和暂无收藏分别提供明确空状态与恢复操作。

## 6. 视觉设计系统

- 字体采用 `-apple-system`、`BlinkMacSystemFont`、`SF Pro Display`、`SF Pro Text` 与适当系统回退；Apple 设备显示 SF 系列，其他设备显示最接近的系统字体。
- 浅色主题使用白色和浅灰背景、深灰正文、克制阴影与半透明材质。
- 深色主题使用接近黑色的背景、分层深灰表面和调亮后的来源色，不使用纯黑大面积压死层次。
- 圆角、阴影、间距和动效遵循统一设计令牌。
- 动效用于卡片进入、悬停、筛选变化和详情展开，并尊重 `prefers-reduced-motion`。
- 正文与控件满足可读对比度；焦点状态清晰可见。

## 7. 响应式设计

- 桌面：杂志式多列布局，顶部工具保持紧凑，文章详情居中显示。
- 平板：减少列数并合并部分筛选控件。
- 手机：单列卡片流，头条卡片不超出首屏，筛选使用底部面板，文章详情占满屏幕。
- 主要手机版验收宽度为 390 像素，同时检查 320 至 430 像素范围无横向溢出。
- 所有核心操作均可触摸，按钮热区不小于 44 像素。

## 8. 容错与安全

- 单个信源失败不会阻断其他信源，失败来源保留上一次成功文章。
- 网络超时和可恢复错误进行有限重试；最终状态写入来源状态文件。
- 前端只显示简洁的异常数量和更新时间，不用技术错误打扰普通阅读。
- 清理 feed HTML，移除脚本、事件属性、危险链接和不允许的标签。
- 外部图片失败时自动回退为来源渐变。
- 无效日期不会伪装为抓取时间。
- 更新器返回非零状态仅限整体数据不可用或输出校验失败；个别信源失败属于可容忍状态。

## 9. 测试与验证

- 单元测试：日期解析、URL 规范化、去重、摘要清理、封面选择、颜色稳定性和数据合并。
- 固定样本测试：RSS 2.0、Atom、带命名空间媒体字段以及普通网页 feed discovery。
- 真实抓取测试：运行全部信源，记录成功、自动发现、失败和保留旧数据的数量。
- 交互测试：搜索、来源与类别筛选、收藏、已读、详情、主题切换和状态持久化。
- 响应式测试：桌面 1440 像素、平板 768 像素和手机 390 像素。
- 无障碍检查：键盘导航、焦点、语义标签、对比度、减少动效和模态焦点管理。
- 发布检查：生产构建、GitHub Actions 手动运行、Pages 地址访问、数据更新时间与外链。

## 10. 验收标准

- 用户提供的全部信源均进入配置，并有可追踪的抓取状态。
- 可用文章按真实发布时间倒序展示，无明显重复。
- 来源颜色在浅色和深色主题下均可辨认。
- 有封面文章显示图片，无封面文章稳定回退为来源渐变。
- 站内摘要详情、阅读原文、收藏、已读、搜索和筛选全部可用。
- 主题支持浅色、深色和跟随系统，刷新后保留手动选择。
- 桌面与手机版无主要内容裁切、横向滚动或不可触摸控件。
- 每日自动更新、手动更新和 GitHub Pages 发布均成功验证。
- 最终交付公开网站地址和公开 GitHub 仓库地址。
